# Web Speed Hackathon Notes

## 調査の進め方

このプロジェクトでは、調査や比較検証をするときは `git branch` を切り、`git worktree` で作業ディレクトリを分けて進める。

- パターンごとに branch を分ける
- branch ごとに worktree を作る
- それぞれを独立して調査する
- 比較は同じ起点から行う

これにより、複数案を同時に検証しても未コミット変更が混ざらず、差分比較もしやすくなる。

## worktree 作成の扱い

調査のために branch / worktree を新しく作る必要がある場合、明示的な指定がなければ勝手に作らない。

必ず次の確認をしてから実施する。

- branch / worktree を作ってよいか
- 何案を並行で比較するか
- 既存の作業ディレクトリに影響を出さないか

## 変更時の基本方針

変更は、レギュレーションに違反しないよう最大限注意して行う。

- レギュレーション違反になりうる変更はしない
- 判断が曖昧なときは、安全側を選ぶ
- パフォーマンス改善のためでも、要件逸脱はしない

## Lighthouse 改善の前提

このプロジェクトでの変更目的は、`Lighthouse` の点数を上げることにある。

ただし、点数を上げるためにユーザーが本来見るべき表示を壊してはいけない。

- 絶対に表示されるべきものは変えない
- 本来表示されるコンテンツを消さない
- 見た目だけ軽く見せるための不正な省略をしない
- 初期表示に必要な情報を不自然に隠さない

「削ぎ落とす」場合でも、表示要件を守ることを前提にする。

## 安全な変更のルール

変更時は次を守る。

- まず計測して、何が重いかを確認してから変える
- 影響が小さい変更から試す
- 1 つずつ変更して差分を確認する
- 変更後は表示、動作、スコアを確認する
- 元に戻せる単位で変更する

## 禁止に近い扱いにするもの

次のような変更は、原則として避ける。

- 表示必須要素の削除
- ユーザー操作に必要な機能の削除
- 採点逃れのためだけの遅延表示
- レギュレーション解釈が怪しい最適化
- 比較不能になる大きすぎる変更の同時投入



`reports/` に次が生成される（`.gitignore` 済み）。

- `bundle-report.html` … treemap（既定は gzip 換算）
- `webpack-stats.json` … webpack stats（CLI・エディタ向け）

## 推奨する進め方

1. 現状を計測する
2. 比較案ごとに branch / worktree を分ける
3. 影響の小さい改善から試す
4. レギュレーション違反がないか確認する
5. Lighthouse と表示内容の両方を確認する
6. 採用案だけを整理して残す

## チェックリスト更新ルール

パフォーマンス改善を進めたら、チェックリストも必ず更新する。

- [all-practice-checklist-template.md](/Users/shimamurayuudai/Documents/hackathon/webspeedhakathon/docs/template/all-practice-checklist-template.md) はテンプレートなので、通常の進行では更新対象にしない
- 改善を実施したら、対応した項目を実ファイル側で記録する
- まだ未着手のもの、判断保留のものも実ファイル側に残して管理する

## 作業報告ルール

作業後の報告では、完全に終わったのか、一部だけ進めたのかを必ず明記する。

- 完全に終わった場合:
  - 何を完了したか
  - チェックリストのどこを更新したか
  - 残作業がないのか、あるなら何が残っているか
- 一部実行の場合:
  - 今回どこを変更したか
  - どのファイルのどのあたりをどれくらい変えたか
  - まだ残っている作業は何か
  - 次にどこを進めればよいか

中途半端に「やった」とだけ書かず、変更範囲と残件が後から見て分かる状態で残す。
