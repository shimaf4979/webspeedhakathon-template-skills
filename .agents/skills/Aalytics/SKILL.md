# Aalytics：パフォーマンス改善の探索順序と重要度

`docs/template/all-practice.md` を元に、分析・改善の「探索する順番」と「重要度（大きく変わりそうなもの）」を定義する。

---

## 重要度の考え方（どれが大きく変わりそうか）

`all-practice.md` の括弧内数字は「何本の記事で言及されたか」。数字が大きいほど、多くの事例で効いた・前提として触れられる施策であり、**見落としているとスコアが大きく変わることが多い**。

### 最優先で見るもの（重要度：高）

施策の有無で **LCP / FCP / TBT / CLS が大きく変わりうる**もの。まずここから確認・改善する。

| 論点 | 記事件数 | なぜ重要か |
|------|----------|------------|
| **Minify / バンドル / Tree Shaking を有効化する** | 13 | 転送量・パースコストに直結。無効だと他をいじっても伸びにくい |
| **必要なデータだけ返す**（API/SSR） | 13 | データ量・Waterfall の根幹。ここが重いとフロントだけでは限界 |
| **軽量なライブラリを選ぶ / 不要部分を削る** | 13 | バンドルサイズと実行コストの大部分を決める |
| **gzip / Brotli / CDN / HTTP キャッシュ** | 12 | 配信レイヤー。効いていないとネットワーク・再訪問が常に重い |
| **不要コード・不要 import を削除する** | 11 | バンドルと実行時間の「無駄」を一気に減らせる |
| **fetchpriority / loading / preload / preconnect 等** | 11 | 読み込み優先度・Waterfall の並びが変わり、LCP/FCP が大きく動く |
| **不要な CSS / JavaScript / DOM を削る** | 10 | 転送・パース・スタイル計算・レイアウトすべてに効く |
| **画像形式・画像サイズの最適化** | 11 | LCP や転送量が一気に変わりやすい |
| **依存をやめて自前実装 / バンドラ・ビルド見直し** | 11 | アーキテクチャレベルの変更で、後続の最適化余地が広がる |
| **SSR / SSG / ISR へ移行する** | 9 | 初期表示の形が変わり、FCP/LCP が桁で良くなることがある |
| **CLS を起こさないようサイズ・アスペクト比を先に確定** | 8 | CLS は「未指定」か「指定済み」で点数が大きく分かれる |

これらは「やっているか・やれているか」の有無でスコアが大きく変わるので、**最初に現状を確認し、抜けがあれば優先して手を入れる**。

---

### 次に効きやすいもの（重要度：中）

記事での言及も多く、**計測・設計の見直しで改善幅が大きくなりやすい**もの。

| 論点 | 記事件数 | なぜ効くか |
|------|----------|------------|
| バンドルサイズを計測する | 10 | 何を削るかの判断材料。計測しないと改善が打ちづらい |
| 再レンダリング範囲を小さく設計する | 10 | TBT/INP に直結。範囲を絞るだけで体感が変わる |
| CSS で表現できるものは CSS に寄せる | 10 | JS・DOM・再レンダリングをまとめて減らせる |
| ユーザー操作後の遅延読み込み | 10 | 初期バンドルと実行コストを減らし、FCP を改善しやすい |
| Tree Shaking を効かせやすい設計 | 7 | 設計次第でバンドルがかなり変わる |
| requestIdleCallback / rAF / Scheduler.postTask | 7 | 長いタスクを分割し、メインスレッドのブロックを減らす |
| 巨大なインライン SVG / base64 埋め込みを避ける | 7 | 1 箇所で HTML/JS が重くなり、LCP を悪化させやすい |
| DB インデックス・クエリのチューニング | 7 | API が遅い原因がここにあると、フロントだけでは限界 |

「まず最優先を潰したあと」に、**計測（バンドル・Core Web Vitals）に合わせて** ここを順に探索する。

---

### 状況依存で効くもの（重要度：状況次第）

記事での言及は少なめだが、**該当する場合に効きが大きい**もの。

- **ServiceWorker / Navigation Preload**（3）：オフライン・再訪問が多いと効く
- **ランタイムのスタイル生成をやめてビルド時生成**（6）：UnoCSS 等のランタイムモードを使っていると効く
- **代替ライブラリへ置き換える**（6）：特定の重いライブラリがあると効果大
- **コンポーネントの遅延読み込み**（6）：ルートや巨大コンポーネントがあると効く
- **重い・危険な正規表現を避ける**（5）：ReDoS や入力時のフリーズがあると効く
- **IntersectionObserver / ResizeObserver**（5）：スクロール連動やリサイズ処理があると効く
- **静的ファイル化**（5）：動的でなくてよいページがあると効く

「該当しそうか」を先に切り分けてから、該当する論点を探索する。

---

## 探索する順番（分析・改善の流れ）

`all-practice.md` のカテゴリを、**計測 → 原因の切り分け → 施策** の順でたどる。

### フェーズ 1：現状把握と「最優先」の確認（まずここから）

1. **計測の準備**
   - Core Web Vitals（LCP, FCP, CLS, INP/TBT）の現状値を取得する
   - バンドルサイズ・リソース数・転送量を把握する（Lighthouse / Bundle 分析）

2. **配信・データ・バンドル基盤（大きく変わりやすい）**
   - カテゴリ **7. サーバー / 配信 / ネットワーク**  
     - 必要なデータだけ返しているか / 余計なレスポースがないか  
     - gzip/Brotli・CDN・HTTP キャッシュが効いているか  
   - カテゴリ **5. ライブラリ / バンドル戦略**  
     - Minify・バンドル・Tree Shaking が有効か  
     - 軽量なライブラリ選定・不要部分削減・バンドルサイズ計測  
   - カテゴリ **2. HTML/CSS/リソース** の「土台」  
     - 不要な CSS/JS/DOM 削除、画像形式・サイズ最適化  
     - fetchpriority / loading / preload / preconnect の有無と妥当性  

3. **CLS と LCP の直接要因**
   - カテゴリ **2** の「CLS を起こさないようサイズ・アスペクト比を確定」「画像の width/height」  
   - LCP 要素の特定と、そのリソースの優先度・preload・画像最適化

ここまでで「大きく変わりそうな」抜けを潰す。

---

### フェーズ 2：JS と読み込みの最適化

4. **カテゴリ 1. JavaScript 実装最適化**
   - 不要 import・不要コード削除（11）
   - Tree Shaking を効かせやすい設計（7）
   - 非同期の並列化・待たなくてよい処理を待たない
   - 重い正規表現の有無（5）

5. **カテゴリ 2. HTML/CSS/リソース（続き）**
   - defer/async、クリティカル CSS の位置、非クリティカルな後ろ倒し
   - ランタイムスタイル生成 → ビルド時生成の可否（6）
   - 巨大インライン SVG/base64 の有無（7）

6. **カテゴリ 3. ブラウザ API 活用**
   - requestIdleCallback / rAF / Scheduler.postTask（7）
   - IntersectionObserver / ResizeObserver（5）
   - ServiceWorker 範囲・Navigation Preload（3）

---

### フェーズ 3：SPA・コンポーネントとアーキテクチャ

7. **カテゴリ 6. SPA / コンポーネント設計最適化**
   - SSR/SSG/ISR の可否（9）
   - 再レンダリング範囲の設計（10）
   - ユーザー操作後の遅延読み込み・コンポーネントの遅延読み込み（10, 6）
   - CSS で表現できるものを CSS に寄せる（10）

8. **カテゴリ 4. ランタイム / JS エンジン**
   - ボトルネックが JS 実行にある場合に、V8 最適化を意識した書き方（型・オブジェクト形状）を検討

9. **カテゴリ 7. サーバー / 配信（深掘り）**
   - DB インデックス・クエリ（7）
   - 圧縮方式・タイミング、103 Early Hints など

---

### 探索順の一覧（短く）

1. **計測**（CWV + バンドル・リソース）
2. **7. サーバー/配信**（データ量・圧縮・CDN・キャッシュ）
3. **5. ライブラリ/バンドル**（Minify/Tree Shaking・ライブラリ・計測）
4. **2. HTML/CSS/リソース**（不要削除・画像・優先度・CLS）
5. **1. JavaScript 実装**（不要削除・Tree Shaking・非同期・正規表現）
6. **2 の続き**（defer/async・スタイル生成・インライン）
7. **3. ブラウザ API**（rIC/rAF・Observer・SW）
8. **6. SPA/コンポーネント**（SSR・再レンダリング・遅延読み込み・CSS に寄せる）
9. **4. ランタイム/V8**（必要に応じて）
10. **7 の深掘り**（DB・圧縮・Early Hints）

---

## この SKILL の使い方

- **分析タスク**：上記の順で「現状やっているか／計測値」を確認し、抜けをリストアップする。
- **改善タスク**：重要度の高いものから順に、「all-practice.md の該当セクション」を参照して具体的なやり方・選択肢を選ぶ。
- **all-practice.md** は詳細な「やり方・向いている場面・注意点」の参照元として使う。
